# Feed Backend — Realtime Coaching Feed API

A lightweight REST + WebSocket backend for a realtime coaching feed. Built with **Express**, **Socket.IO**, **PostgreSQL**, and **Redis**.

## Overview

This service manages a chronological feed of coaching posts. Any connected client receives new posts in realtime via WebSocket. Read endpoints are cached in Redis to reduce database load. Write endpoints are admin-only, protected by JWT.

---

## Assumptions

1. **Single feed, no user accounts** — There is one global feed. The service does not handle user registration or login. Admin tokens are pre-issued (or generated via the dev helper endpoint).

2. **Admin-only writes** — Creating, updating, and deleting feed items requires a JWT with `role: "admin"`. Regular consumers only read.

3. **Stateless auth** — Authentication is JWT-based with a shared secret (`ADMIN_JWT_SECRET`). No token revocation or refresh flow is implemented; tokens are short-lived by convention.

4. **Offset pagination, not cursor** — The feed is paginated with `page` + `limit` query parameters. This is simpler to implement and sufficient for moderate feed sizes. Cursor-based pagination would be preferable at very high volume.

5. **Aggressive cache invalidation** — Any write (POST, PUT, DELETE) wipes all `feed:list:*` cache keys. This keeps the cache simple and always consistent at the cost of a brief cache-cold period after writes.

6. **No media attachments** — Feed items contain only text fields (`title`, `content`, `author`). File/image upload is out of scope.

7. **No soft deletes** — Deleted rows are permanently removed from the database.

8. **Client-side reconnect recovery** — Clients that disconnect and reconnect can emit a `sync` Socket.IO event with their `lastSeenAt` timestamp to receive any missed posts. The server does not track per-client state.

9. **PostgreSQL manages IDs and timestamps** — `id` is a UUID generated by `pgcrypto`'s `gen_random_uuid()`. `created_at` and `updated_at` are managed by the database via a trigger, not the application layer.

10. **Dev token endpoint is not production-safe** — `GET /dev/token` is only available when `NODE_ENV !== "production"` and exists purely for local testing convenience.

## Architecture & Design

```
Client (REST / WebSocket)
        │
        ▼
  Express HTTP Server  ──────────────  Socket.IO (same HTTP server)
        │                                      │
        │  /api/feed CRUD                      │  feed:new  (broadcast on POST)
        │                                      │  feed:catchup  (on sync event)
        ▼                                      │
   Auth Middleware (JWT)                       │
        │                                      │
        ▼                                      │
   Feed Router ─────── Redis (cache) ──────────┘
        │
        ▼
   PostgreSQL (source of truth)
```

**Request flow for GET /api/feed:**
1. Check Redis for a cached response keyed by `feed:list:{page}:{limit}`.
2. Cache hit → return immediately, marking `meta.source = "cache"`.
3. Cache miss → query PostgreSQL, store result in Redis with TTL, return to client.

**Request flow for POST/PUT/DELETE /api/feed:**
1. Validate JWT (`requireAdmin` middleware).
2. Validate request body (`express-validator`).
3. Execute the database mutation.
4. Invalidate all `feed:list:*` keys in Redis via `SCAN` + `DEL`.
5. On POST: broadcast the new row to all Socket.IO clients as `feed:new`.

## Database Schema

```sql
-- Extension for UUID generation
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

CREATE TABLE feeds (
  id          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  title       VARCHAR(200) NOT NULL,
  content     TEXT         NOT NULL CHECK (char_length(content) <= 5000),
  author      VARCHAR(100) NOT NULL,
  created_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

-- Index for efficient chronological reads
CREATE INDEX idx_feeds_created_at ON feeds (created_at DESC);

-- Trigger: auto-update updated_at on every row update
CREATE TRIGGER feeds_updated_at
  BEFORE UPDATE ON feeds
  FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

Run the migration with:

```bash
npm run migrate
```

## API Reference

### Base URL

```
http://localhost:4000
```

Health check.

**Response**
```json
{ "status": "ok" }
```

### `GET /api/feed`

Fetch a paginated list of feed items, newest first.

**Query Parameters**

| Param | Type | Default | Description |
|---|---|---|---|
| `page` | integer | `1` | Page number (1-indexed) |
| `limit` | integer | `20` | Items per page (max `100`) |

**Response `200`**
```json
{
  "data": [
    {
      "id": "uuid",
      "title": "string",
      "content": "string",
      "author": "string",
      "created_at": "ISO8601",
      "updated_at": "ISO8601"
    }
  ],
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 42,
    "source": "db" | "cache"
  }
}
```

---

### `POST /api/feed`

Create a new feed item. **Requires admin JWT.**

**Headers**
```
Authorization: Bearer <token>
Content-Type: application/json
```

**Body**
```json
{
  "title":   "string (max 200)",
  "content": "string (max 5000)",
  "author":  "string (max 100)"
}
```

**Response `201`**
```json
{ "data": { ...feedItem } }
```

**Side effects:** Clears all Redis feed cache keys. Broadcasts `feed:new` to all Socket.IO clients.

---

### `PUT /api/feed/:id`

Update an existing feed item. **Requires admin JWT.** At least one of `title`, `content`, or `author` must be provided.

**Headers**
```
Authorization: Bearer <token>
Content-Type: application/json
```

**Body** (all fields optional, but at least one required)
```json
{
  "title":   "string (max 200)",
  "content": "string (max 5000)",
  "author":  "string (max 100)"
}
```

**Response `200`**
```json
{ "data": { ...updatedFeedItem } }
```

**Response `404`** — Feed item not found.

**Side effects:** Clears all Redis feed cache keys.

---

### `DELETE /api/feed/:id`

Permanently delete a feed item. **Requires admin JWT.**

**Headers**
```
Authorization: Bearer <token>
```

**Response `200`**
```json
{ "message": "Deleted successfully" }
```

**Response `404`** — Feed item not found.

**Side effects:** Clears all Redis feed cache keys.

---

### `GET /dev/token` _(development only)_

Generates a signed admin JWT valid for 7 days. Only available when `NODE_ENV` is not `"production"`.

**Response `200`**
```json
{ "token": "<jwt>" }
```

## WebSocket Events

Connect to `ws://localhost:4000` using a Socket.IO v4 client.

### Events emitted **by the server**

| Event | Payload | Description |
|---|---|---|
| `feed:new` | `{ eventId: uuid, data: feedItem }` | Broadcast to all clients when a new feed item is created |
| `feed:catchup` | `{ data: feedItem[] }` | Sent to a single client in response to a `sync` event; contains all items created after `lastSeenAt` |
| `error` | `{ message: string }` | Sent to the requesting client when a socket operation fails |

### Events emitted **by the client**

| Event | Payload | Description |
|---|---|---|
| `sync` | `{ lastSeenAt: ISO8601 }` | Send on reconnect to receive any items missed while disconnected |

**Example (browser)**
```js
import { io } from 'socket.io-client';

const socket = io('http://localhost:4000');

socket.on('feed:new', ({ data }) => {
  console.log('New post:', data);
});

// On reconnect, recover missed items
socket.on('connect', () => {
  socket.emit('sync', { lastSeenAt: localStorage.getItem('lastSeenAt') });
});

socket.on('feed:catchup', ({ data }) => {
  console.log('Missed posts:', data);
});
```

## Caching Strategy

- All `GET /api/feed` responses are cached in Redis under the key `feed:list:{page}:{limit}`.
- TTL defaults to **300 seconds** (configurable via `CACHE_TTL_SECONDS`).
- Any write operation (POST / PUT / DELETE) immediately invalidates **all** `feed:list:*` keys using a Redis `SCAN` loop to avoid using `KEYS` in production.
- The `meta.source` field in GET responses indicates whether the response came from `"db"` or `"cache"`.

---

## Authentication

Write endpoints use **Bearer JWT** authentication.

- Tokens must be signed with `ADMIN_JWT_SECRET` and carry the claim `"role": "admin"`.
- The `requireAdmin` middleware validates the token and rejects requests with a missing role or invalid signature.
- In development, call `GET /dev/token` to mint a 7-day admin token without any additional tooling.

```bash
# Get a dev token
TOKEN=$(curl -s http://localhost:4000/dev/token | node -e "process.stdin.resume();process.stdin.on('data',d=>console.log(JSON.parse(d).token))")

# Use it
curl -X POST http://localhost:4000/api/feed \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"title":"Hello","content":"First post","author":"Coach A"}'
```

## Getting Started

### Prerequisites

- Node.js 18+
- PostgreSQL 14+
- Redis 7+

### 1. Install dependencies

```bash
npm install
```

### 2. Configure environment

```bash
cp .env.example .env
# Edit .env with your database URL, Redis URL, and JWT secret
```

### 3. Run database migration

```bash
npm run migrate
```

### 4. Start the server

```bash
# Development (auto-restart on file changes)
npm run dev

# Production
npm start
```

The server starts on `http://localhost:4000` (or the port in `PORT`).
